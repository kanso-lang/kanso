chapter 03

types and dispatch

every type in kanso is a single-constructor record: a name and some fields. the fields are alphabetical — enforced, like everything else other languages leave to code review. construction is positional, using the type name as the constructor, and taking a record apart is a binding pattern — write the constructor shape on the left of = and every field gets a name:

records.kso
type track
  artist: string
  minutes: int
  title: string

main =
  song = track "fishmans" 6 "long season"
  track artist minutes title = song
  print "{title}, by {artist} — {minutes} minutes"
output
long season, by fishmans — 6 minutes

the positional form binds every field, which is honest but sometimes more than you want. when you only care about a couple of fields, use the keyed read — name the fields you want, rename freely with :, skip the rest:

keyed.kso
type track
  artist: string
  minutes: int
  title: string

main =
  song = track "fishmans" 6 "long season"
  { artist: who title } = song
  print "{title}, by {who}"
output
long season, by fishmans

if you try to split the difference — positional destructuring with _ for the fields you'd rather not name — the compiler redirects you to the right tool:

peek.kso
type track
  artist: string
  minutes: int
  title: string

main =
  song = track "fishmans" 6 "long season"
  track artist _ title = song
  print "{title}, by {artist}"
kanso check peek.kso
error[syntax]: `_` does not appear in binding patterns; omit fields with a keyed read
  --> peek.kso:8:16
   8 |   track artist _ title = song
                      ^

the third place records come apart is in a function head, where _ is welcome — a parameter pattern both selects the type and unpacks it in one gesture:

credit.kso
type track
  artist: string
  minutes: int
  title: string

fn credit (track artist _ title)
  "{title} — {artist}"

main =
  song = track "fishmans" 6 "long season"
  print (credit song)
output
long season — fishmans

the only switch

that function head is doing something bigger than convenience. kanso has no match, no instanceof, no tag tests, no narrowing syntax. a value may inhabit a typeset — it might be an int or a string, a result or an err — and the only way to take that apart is to define overloads and let dispatch choose. the conditional you would have written lives in the function head instead:

describe.kso
fn describe 0
  "zero"

fn describe n:int
  "the int {n}"

fn describe s:string
  "the string {s}"

fn describe x
  "something else: {x}"

main = print (describe 0) >> print (describe 7) >> print (describe "seven") >> print (describe 1.5)
output
zero
the int 7
the string seven
something else: 1.5

four arms, one name, and a specificity ladder deciding who answers: a literal like 0 beats an ascribed concrete type like n:int, which beats the bare generic x. (note the ascription is tight — n:int, no spaces; n: int is a formatting error.) constructor patterns like (track artist _ title) sit with the concrete types on this ladder, which is exactly what makes them the way to handle err in the next chapter.

dispatch on literals is what turns the classic recursive definition into something you can read aloud — the base case is an overload, not an if guarding the real body:

fact.kso
fn fact 0
  1

fn fact n
  n * fact (n - 1)

main =
  answer = fact 20
  print "20! = {answer}"
output
20! = 2432902008176640000

that answer doesn't fit in 64 bits and nobody had to care — int is arbitrary-precision, as promised.

the ladder is law

because dispatch replaces every switch you'd otherwise write, kanso is strict about how overload groups are laid out and what they may contain. the arms appear most-specific first, so the file reads like the dispatcher resolves:

backwards.kso
fn fact n
  n * fact (n - 1)

fn fact 0
  1
kanso check backwards.kso
error[formatting]: overloads of `fact` appear most-specific first: literal, then concrete type, then generic
  --> backwards.kso:4:4
   4 | fn fact 0
          ^

and two arms that dispatch can never tell apart are rejected outright, because a switch with two identical cases is a bug wherever it appears:

twins.kso
fn same x
  x

fn same y
  y
kanso check twins.kso
error[dispatch]: overlapping overloads of `same` are illegal
  --> twins.kso:6:4
   6 | fn same y
          ^

for the honest cases where a value is just a boolean decision, if exists — but as a plain function, if condition then_value else_value, an expression like everything else. no statement form, no dangling else, no ternary syntax to argue about:

admission.kso
fn admission age
  if (age < 13) "child ticket" "regular ticket"

main = print (admission 9) >> print (admission 30)
output
child ticket
regular ticket

the rule of taste: if is for decisions about one value's content; overloads are for decisions about a value's shape. when you find yourself asking "which kind of thing is this?", the answer is never a conditional — it's another arm.

ensō-neko

fields alphabetical, arms most-specific first, ascription tight as n:int. there is exactly one place everything goes, which means you never have to decide — and neither does the person reading after you.

tsuru

a note for later: those overload arms are not tag checks at runtime. i resolve dispatch statically per instantiation, and literal arms on one parameter become a jump table. chapter 07 has the ir to prove it.
