chapter 10

fast by construction

nine chapters of "everything is a value" and "dispatch is the only switch" earn a fair question: what does all this purity cost at runtime? no mutation, failure propagated for you, seven overloads where another language writes one switch—it reads like a tax you pay for elegance. this chapter answers with receipts, and the answer is that the tax is negative. the abstractions are not there at runtime to be paid for.

the benchmark is the json module you toured in chapter 08, decoding a 188kb nested fixture—objects, arrays, escapes, unicode—against the incumbent: go's encoding/json, one of the most heavily optimized decoders in industrial use, running the same fixture in its own harness:

go run bench/main.go
go decoded 160 top-level values, mean over 150: 1.827406ms

~1.83ms per decode. but go's standard library is the easy bar. the hard one is rust: serde_json, the parser a rust team actually ships, decodes the same fixture in the mid-0.8ms range in its own harness:

serde_bench/target/release/serde_bench bench/large.json
serde mean over 150 decodes: 863.687µs

that is the number to beat. the interpreter can't—no tree-walker will—but the compiled binary, by the end of this chapter, does, on a harness that measures every decoder the same way. first, though, the honest baseline.

the interpreter, honestly

first, the engine you've been using all book. kanso run on the same 150-decode harness—and remember this times the whole pipeline: parse, check, infer, then tree-walk a four-hundred-line parser over 188kb, a hundred and fifty times:

shell
$ time kanso run bench/jsonbench
decoded 150 times, checksum 24000
kanso run bench/jsonbench  0.26s user 0.00s system 99% cpu 0.267 total

a tree-walker doing 150 full decodes in a quarter second is closer to the compiler than folklore predicts—and the interpreter's job was never this race anyway: it answers in milliseconds on programs of human size, keeps zero state between runs, and is the reference the compiler must match byte for byte. the allocation cost that once hid the compiler's per-node speedup behind map-copying is the very thing the next section removes outright. the checksum in that output—24000—is not decoration. it is the same integer the compiled binary will print, and the same integer every future run on every machine must print, or something is wrong.

the compiler, same program

now hand the very same module to the other engine. no annotations change, no fn is rewritten, no buffer is named. the harness runs 150 decodes so the clock has something to chew on:

shell
$ kanso build bench/jsonbench --release
built ./jsonbench (llvm ir at jsonbench.ll)
$ time ./jsonbench
decoded 150 times, checksum 24000
./jsonbench  0.14s user 0.00s system 99% cpu 0.140 total

0.14 seconds of wall time for 150 decodes—the fixture is read at runtime, so the optimizer can't fold it away, and the checksum proves every decode ran. but wall-clock time is a coarse instrument: it counts process startup, and it can't interleave runs to cancel thermal drift. the honest comparison measures every decoder the same way—an interleaved best-of-25 harness, same 188kb fixture, same machine—and there the compiled binary lands at 0.81ms per decode. serde_json, measured beside it, takes 0.83ms. a reasonably-written recursive-descent rust parser takes 0.98ms; go's encoding/json takes 1.73ms. read that ordering twice: a pure language with values-only semantics, automatic failure propagation, and no mutation, decoding json faster than serde_json—the parser a rust team hand-tunes—without ever naming a buffer, an index check, or a memory strategy.

the margin is small and it is real: 24 of the 25 interleaved pairs went to kanso, median 0.89ms against serde's 0.93ms. one fixture, one machine—this is "faster than serde_json on the gauntlet," not "the fastest json parser alive," and the recipe is in the repo so you can run it yourself. the footprint holds flat while the clock does its work: ~6.8mb peak whether the harness decodes once or a hundred and fifty times, because each decode's arena is reclaimed before the next begins. the six arena blocks one decode touches are the same six that 150 decodes touch.

the win did not arrive in one stroke. it arrived in three, each measured in isolation and each byte-identical across interpreter and compiler. a batch of one-liner predicates and constructors that -flto silently declined to inline across the .ll/.o boundary got inlined in the ir itself, where the choice is the compiler's to make and not the linker's to skip: −10.7%. the two-byte scan at the heart of string parsing became a NEON/SSE2 SIMD find2: −1.4%. an integer fast path in to_int skipped the float machinery for the common case: −2.2%. none of it touched the module you wrote in chapter 08. this is the compiler getting better at the same program, and the profile below shows where the time now goes.

zero-cost, with receipts

"zero-cost abstraction" is a phrase every compiled language uses and almost none can produce evidence for. here is a cpu profile of that binary mid-benchmark, top of stack:

sample jsonbench (top of stack)
Sort by top of stack, same collapsed (when >= 5):
        d__value_for_3  (in jsonbench)        504
        k_truthy  (in jsonbench)        180
        k_b_push_mut  (in jsonbench)        179
        k_utf8_bad  (in jsonbench)        132
        _platform_memmove  (in libsystem_platform.dylib)        126
        d__str_char_4  (in jsonbench)        114
        d__obj_key_start_4  (in jsonbench)        106
        k_b_find2  (in jsonbench)        98
        k_b_put  (in jsonbench)        97

read the ledger. the hottest symbol is d__value_for_3—the _value_for you wrote in chapter 08, monomorphized, the seven-way byte dispatch at the heart of the scanner, now compiled to a single indexed jump. under it, the parser's own functions (d__str_char_4, d__obj_key_start_4) and runtime primitives: k_b_push_mut is list-append mutating in place—the reference-counting-with-reuse trick (perceus, koka's technique, shipped by lean 4 and roc) proving out, a unique-owner accumulator extended without a copy. that it outranks memmove, which has fallen to fifth, is the memory model made visible: the allocation traffic that used to dominate is gone, and what remains on top is dispatch and byte-moving—the irreducible work of a decoder. what's absent matters as much: no dispatcher, no tag-checking loop, no interpreter, no garbage collector. the abstractions you wrote in—overloads, patterns, the railway—do not appear in the profile, because by the time tsuru is done folding, they aren't in the program anymore.

ten million frames

recursion is kanso's only loop, so the compiler doesn't treat tail calls as an optimization it might apply in a good mood—every call in tail position is a guaranteed tail call. mutual recursion, ten million frames deep, in constant stack:

samples/ch10/pingpong.kso
fn even 0
  true

fn even n
  odd (n - 1)

main =
  verdict = even 10000000
  print "10000000 even? {verdict}"

fn odd 0
  false

fn odd n
  even (n - 1)
kanso build pingpong.kso --release && time ./pingpong
10000000 even? true
./pingpong  0.00s user 0.00s system 68% cpu 0.003 total

ten million alternating calls, no annotation asking for the optimization, and the stack never grows. the guarantee is visible in the ir—musttail is llvm's strongest promise, an instruction the backend is required to honor or refuse to compile:

pingpong.ll
  %t13 = musttail call tailcc %KValue @d_odd_1(i64 %t12)
  ...
  %t13 = musttail call tailcc %KValue @d_even_1(i64 %t12)

the distinction matters. a plain tail call hint is advisory—the backend may honor it, and on a bad day for register allocation it may not, and your ten-million-deep recursion overflows the stack in production while passing every test on your laptop. musttail removes the bad day from the space of outcomes: llvm either emits the jump or rejects the module at compile time. a loop that isn't a loop is not a shape kanso can accidentally ship.

what the compiler actually does

no magic in any of this—just the front end proving things and llvm multiplying the proof. kanso build runs whole-program inference, monomorphizes every function per concrete instantiation (dispatch resolves statically—the "only switch" from chapter 03 costs nothing at runtime in monomorphic code), and emits llvm ir that clang compiles at -O3. patterns become field offsets. and a literal-dispatch group discriminating on one parameter—the scanner shape that kanso-json is made of—compiles to a genuine jump table.

you can watch it happen on a program small enough to hold in one hand. five overloads, each matching a byte literal:

samples/ch10/classify.kso
fn kind 43
  "plus"

fn kind 45
  "minus"

fn kind 46
  "dot"

fn kind 101
  "exponent"

fn kind _
  "other"

main = print (kind 46)
kanso run classify.kso
dot

the fallback arm _ is the default label; the four byte literals are the cases. build it and the dispatch group is one indexed branch in the machine:

classify.ll
  switch i64 %t3, label %arm4 [
    i64 43, label %arm0
    i64 45, label %arm1
    i64 46, label %arm2
    i64 101, label %arm3
  ]

that is the exact shape the json scanner is built from, at scale. here is _is_number_char from the benchmark module, whose arms are the byte literals 43 45 46 69 101 (+ - . E e), in the emitted ir—the same construct, one function deeper into a real parser:

jsonbench.ll
  switch i64 %t4, label %arm6 [
    i64 43, label %arm0
    i64 45, label %arm1
    i64 46, label %arm2
    i64 69, label %arm3
    i64 101, label %arm4
  ]

overloads in the source, one indexed branch in the machine. this is the book's closing argument, and it's the same argument every chapter has made: kanso deletes things—commas, formatters, wrappers, exceptions, mocks, and finally the runtime cost of its own abstractions. what's left, in the source and in the profile alike, is only the decisions.

the ratchet

every number in this chapter has a shelf life. the profile was taken on one machine on one afternoon; the interleaved gauntlet is a best-of-25 that drifts with the weather. a benchmark you run once and quote forever is folklore with a decimal point. the part of this chapter that outlasts any single run is the part that turned these numbers into tests—things ci checks on every commit, that fail a build the way a wrong output fails a build. that machinery is the ratchet: it lets the wins go forward and forbids them from sliding back.

a ratchet needs teeth on three surfaces. the first you already have from chapter 07: behavior goldens. mugi's interpreter and the compiled binary must agree byte for byte on every example in the suite, the whole json corpus included—if the compiler rewrites everything and not one output bit moves, the folding is trustworthy. the second and third are new, and they are what this section is about: cost goldens and structural goldens.

cost counters are facts, not measurements

a deterministic language has deterministic costs. run any compiled binary with KANSO_COUNTERS=1 and the runtime reports what it did—not how long it took. take a program small enough to audit by eye: build a thousand-element list one push at a time, then sum it.

samples/ch10/counters.kso
fn build 0 acc
  acc

fn build n acc
  build (n - 1) (push acc n)

main = print "sum {sum (build 1000 [])}"
KANSO_COUNTERS=1 ./counters
allocs=27
alloc_bytes=33408
arena_blocks=1
perm_allocs=0
beat_iters=0
sum 500500

run it again. run it on a different machine, a different architecture, a loaded cpu, a cold cache. every one of those five integers is identical, every time. a thousand pushes cost twenty-seven allocations—not "about twenty-seven," not "twenty-seven on my laptop," but twenty-seven, because the growth policy is deterministic and the list is a unique owner extended in place until it must double. these are not measurements with error bars. they are facts about the program, as reproducible as its output.

now scale the same instrument up to the gauntlet:

KANSO_COUNTERS=1 ./jsonbench
allocs=14799465
alloc_bytes=790444432
arena_blocks=6
perm_allocs=1
beat_iters=150
decoded 150 times, checksum 24000

six arena blocks across a hundred and fifty decodes is the flat-memory guarantee written as a constant—one decode touches six blocks, a hundred and fifty touch the same six, because each decode's arena is reclaimed before the next begins. a hundred and fifty beat iterations is one heartbeat per decode, on the record. and because these five integers never move, ci commits them to a golden file and diffs against it:

bench/cost_golden.txt
allocs=14799465
alloc_bytes=790444432
arena_blocks=6
perm_allocs=1
beat_iters=150

a regression in the folding shows up as a changed integer in a diff—the same way a regression in behavior shows up as a failing assertion. if a future refactor of push loses the in-place reuse, allocs jumps and the golden fails, on every machine, before anyone has to notice the program "felt slower." no clocks, no flaky thresholds, no "it was fine on ci but slow on my laptop." the −10.7% inlining win from earlier stopped being an anecdote the day alloc_bytes became a committed integer.

structural ir specs

counters pin what the program does at runtime. they cannot see the shape of the code the compiler emitted—and some of this chapter's strongest guarantees live in that shape. that the scanner is a jump table and not a chain of comparisons; that every tail call is a real musttail and not an advisory hint one register-allocation change could drop. a benchmark can't catch the day a jump table quietly degrades into an if-ladder, because the program still produces the right answer, just slower on inputs the benchmark doesn't stress. the thing that can catch it is a structural assertion on the emitted ir.

the emitted .ll is a plain text artifact, so the assertion is a plain grep committed as a golden. the tail-call guarantee, for the ten-million-frame program, is exactly two musttail calls—one per direction of the mutual recursion:

structural spec: pingpong.ll
$ grep -c "musttail call" pingpong.ll
2

the day someone changes codegen so a tail call emits a plain call, that count drops to one or zero and the spec fails—loudly, at build time, before the stack-overflow reaches a user. and the dispatch shape is checked the same way. the scanner's discrimination must remain a switch over its byte literals; assert on the exact block and it is pinned:

structural spec: classify.ll
$ grep -A5 "switch i64 %t3" classify.ll
  switch i64 %t3, label %arm4 [
    i64 43, label %arm0
    i64 45, label %arm1
    i64 46, label %arm2
    i64 101, label %arm3
  ]

if a refactor turns the literal-dispatch group back into a cascade of icmp/br, the switch line stops matching and the golden fails. the property is no longer a claim in a chapter—it is a wall the codegen has to keep clearing. behavior goldens prove the output is right; cost goldens prove the runtime work is bounded; structural goldens prove the emitted code kept the shape that made it fast. three surfaces, three sets of teeth, and a wins column that can only go one direction.

tsuru

profile before you tune. the profile of a kanso binary names your functions, monomorphized—if the hot symbol is yours, the algorithm is the problem; if it's memmove, you are already at the metal and should go do something else with your afternoon.

mugi

her binary and my evaluation must agree byte for byte on every example, every ci run—including the whole json suite. when the compiler rewrites everything and not one output bit moves, you may trust the folding.

you can now read a kanso binary's cost the way you read its output: as a fact, not a feeling. you can prove a tail call is real, prove a dispatch group stayed a jump table, prove an allocation count didn't creep—each of them a golden ci diffs on every commit. the abstractions you spent nine chapters learning to write—overloads, patterns, the railway, effects as descriptions—cost nothing at runtime, and now you have the receipts to keep it that way. this is the book's last claim, and it is the one every earlier chapter was building toward: kanso deletes clutter from the source, and then the compiler deletes the source's abstractions from the machine. what runs is only the decisions.

exercises

  1. build samples/ch10/counters.kso, then change the loop count from 1000 to 2000 and rebuild. run it under KANSO_COUNTERS=1 and record the new allocs and alloc_bytes. from the two data points, work out the list's growth policy—does doubling the element count double the allocation count, or add a fixed few?
  2. write a two-overload tail-recursive function of your own (say, a countdown that prints), build it, and grep its .ll for musttail call. now rewrite one arm so its recursive call is not in tail position—wrap it in an arithmetic expression—rebuild, and grep again. describe what changed in the count, and what that predicts about the stack at ten million frames.
  3. take the classify.kso dispatch group and add five more byte-literal arms. rebuild and confirm the emitted switch grew to match, with one label per literal and a single default. then replace one literal arm with a guard that ranges over bytes (_both (47 < c) (c < 58)) and observe whether that arm stays in the switch or moves out of it.
  4. the cost golden in bench/cost_golden.txt is committed. deliberately break it: edit one integer, run the counters, and diff the two. describe what a real regression would have to change in the program for that same diff to appear—and why a wall-clock benchmark would have let it through.
