chapter 06

modules and tests

a kanso module is a directory. that's the whole definition. the files inside share one namespace—no import statements between them, no export lists, no manifest declaring what's where. split a module across files for the reader's benefit, and the compiler treats the result as one program:

teahouse/menu.kso
fn _base_price "dango"
  350

fn _base_price "taiyaki"
  500

fn _base_price item
  err "we do not serve {item}"

fn price item
  _base_price item + 50
teahouse/main.kso
fn describe (err reason)
  "gomen: {reason}"

fn describe n
  "{n} yen, service included"

main =
  dango = price "dango"
  pocky = price "pocky"
  print (describe dango) >> print (describe pocky)
kanso run teahouse
400 yen, service included
gomen: we do not serve pocky

point the toolchain at the directory and it runs the module. main.kso calls price from menu.kso as if it were next door, because it is. notice also three chapters shaking hands in twelve lines: string-literal dispatch is the menu, an err is the missing item, and the railway carries it from _base_price through price—whose + 50 never runs on the sad track—to the arm in describe.

underscore is privacy

the leading underscore on _base_price is not a naming fashion; it's the visibility system. an underscored name is module-private—the module's own business, invisible as api. everything else is public. there is no pub keyword, no export list to maintain, no visibility matrix to memorize: the name itself tells every reader, at every call site, whether they're looking at surface or interior. and because a private name with no callers is waste, kanso flags an unused private declaration the same way it flags an unused binding.

tests are constants

chapter 01 promised that tests are just test_ constants. drop a test file into the module—it shares the namespace, so it can see private names too, which is occasionally exactly what a test wants:

teahouse/menu_test.kso
fn _is_err (err _)
  true

fn _is_err _
  false

test_dango_price = (price "dango") == 400

test_unknown_item_fails = _is_err (price "pocky")
kanso test teahouse
test_dango_price ... ok
test_unknown_item_fails ... ok
2 passed, 0 failed

a test is a boolean constant; the runner evaluates each one and reports. when a test fails, the report says what the value was, because that's all a failure is:

double_test.kso
fn double n
  n * 2

test_double = (double 4) == 9
kanso test double_test.kso
test_double ... FAILED (returned false)
0 passed, 1 failed

no assertion library, no matcher dsl, no test lifecycle. the entire testing framework is: name a boolean test_something. it turns out that's enough to test a json parser.

a real module: kanso-json in six files

the repository ships lib/json—a complete json decoder and encoder, escapes and \uXXXX and failure positions included, written to prove the language can earn a living. it's six files, four hundred and change lines, and it reads like a table of contents:

  • json.kso—the public api: decode, encode, the failure types
  • value.kso—parsing values: objects, arrays, literals
  • text.kso—strings and escapes
  • number.kso—ints, floats, and which bytes may start them
  • scan.kso—whitespace and low-level cursor work
  • json_test.kso—the suite

one namespace, so value.kso calls scan.kso's helpers without ceremony; underscores everywhere, so the public surface is exactly decode, encode, and the types. and the heart of the parser is the chapter-03 move at industrial scale—the next byte is the dispatch:

lib/json/value.kso
# value starts by byte: 34 quote, 91 bracket, 102 f, 110 n, 116 t, 123 brace
fn _value_for 34 cs p
  _parse_string cs (p + 1)

fn _value_for 91 cs p
  _parse_array cs (p + 1)

fn _value_for 102 cs p
  _word cs p _bytes_false false

fn _value_for 110 cs p
  _word cs p _bytes_null json_null

fn _value_for 116 cs p
  _word cs p _bytes_true true

fn _value_for 123 cs p
  _parse_object cs (p + 1)

fn _value_for none _ p
  _fail p "unexpected end of input"

fn _value_for c cs p
  if (_is_number_start c) (_parse_number cs p) (_fail p "unexpected character `{utf8 [c]}`")

eight arms, one per way a json value can begin—including the none arm, because running off the end of the input is just an absent byte, dispatched like everything else. no scanner state machine, no switch statement, no lookahead abstraction. and the parser contains not one line of "check if the last step failed": a bad escape or truncated input becomes an err deep inside, and the railway carries it out of decode with its position intact.

the suite pins all of it—decoding, encoding, escapes, unicode, error positions, a round trip:

lib/json/json_test.kso
test_decode_unicode = (decode "\"\\u00e9\"") == "é"

test_roundtrip =
  text = "\{\"a\":[1,2.5,\"x\"],\"b\":null}"
  (encode (decode text)) == text

sixteen booleans, one command, and a library that's about to race go. which brings us to the last chapter.

ensō-neko

within each file, declarations stay alphabetical—types first, then everything else. split files by topic, the way kanso-json does, and alphabetical order inside each one stops being a constraint and starts being an index.

mugi

a test is a value, so everything you know about values applies: no setup, no teardown, no ordering, no shared state to leak between examples. sixteen constants cannot flake. i have checked.
