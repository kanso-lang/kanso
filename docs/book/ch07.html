chapter 07

fast by construction

six chapters of "everything is a value" and "dispatch is the only switch" earn a fair question: what does all this purity cost at runtime? this chapter answers with receipts. the benchmark is the json module you toured in chapter 06, decoding a 188kb nested fixture—objects, arrays, escapes, unicode—against the incumbent: go's encoding/json, one of the most heavily optimized decoders in industrial use, running the same fixture in its own harness:

go run bench/main.go
go decoded 160 top-level values, mean over 150: 1.827406ms

~1.83ms per decode. but go's standard library is the easy bar. the hard one is rust: serde_json, the parser a rust team actually ships, does the same fixture in ~0.86ms—and that is the number this chapter is honest about not beating yet.

serde_bench/target/release/serde_bench bench/large.json
serde mean over 150 decodes: 863.687µs

the interpreter, honestly

first, the engine you've been using all book. kanso run on the same 150-decode harness—and remember this times the whole pipeline: parse, check, infer, then tree-walk a four-hundred-line parser over 188kb, a hundred and fifty times:

shell
$ time kanso run bench/jsonbench
decoded 150 times, checksum 24000
kanso run bench/jsonbench  0.39s user 0.05s system 99% cpu 0.442 total

a tree-walker doing 150 full decodes in under half a second is closer to the compiler than folklore predicts—and the honest reason is unflattering to both engines: they spend most of their time in the same place, copying growing maps (the memmove the profiler flags), so the compiler's per-node speedup is partly hidden behind an allocation cost neither engine has fixed yet. that changes when reuse analysis ships. either way, the interpreter's job was never this race: it answers in milliseconds on programs of human size, keeps zero state between runs, and is the reference the compiler must match byte for byte.

the compiler, same program

now hand the very same module to the other engine. the harness runs 150 decodes so the clock has something to chew on:

shell
$ kanso build bench/jsonbench --release
built ./jsonbench (llvm ir at jsonbench.ll)
$ time ./jsonbench
decoded 150 times, checksum 24000
./jsonbench  0.23s user 0.04s system 97% cpu 0.277 total

0.27 seconds of wall time for 150 decodes (the fixture is read at runtime, so the optimizer can't fold it, and the checksum proves every decode ran): ~1.8ms per decode. that edges go's ~1.83ms—a pure language with values-only semantics, automatic failure propagation, and no mutation matches go's standard library from source that never mentions a buffer, an index check, or a memory strategy. but serde is still twice as fast, and pretending otherwise would make this whole chapter untrustworthy.

the gap is not the abstractions—it is the allocation. the cpu profile shows memmove and map insertion on top, because building an n-key object still copies the whole map on every put. that is exactly what the deepest type-driven optimizations delete: reference-counting with reuse (perceus—koka's technique, shipped by lean 4 and roc) turns unique-owner updates in place, and the compiler page stages the campaign to catch serde. the abstractions are already free; the memory strategy is the next chapter of the story, not this one.

zero-cost, with receipts

"zero-cost abstraction" is a phrase every compiled language uses and almost none can produce evidence for. here is a cpu profile of that binary mid-benchmark, top of stack:

sample jsonbench (top of stack)
Sort by top of stack, same collapsed (when >= 5):
        _platform_memmove  (in libsystem_platform.dylib)        25
        d__skip_ws_2  (in jsonbench)        14
        k_b_put  (in jsonbench)        14
        k_b_push  (in jsonbench)        12
        d__string_at_4  (in jsonbench)        11
        d__parse_number_2  (in jsonbench)        10
        d__value_for_3  (in jsonbench)        10

read the ledger. the hottest symbol is memmove—the program is spending its time moving bytes, which is what a decoder is for. everything else is either a runtime primitive (k_b_put, k_b_push—map and list construction) or the parser's own functions, monomorphized: d__skip_ws_2, d__value_for_3 are the _skip_ws and _value_for you read in chapter 06, compiled into type-specialized copies. what's absent is the tell: no dispatcher, no tag-checking loop, no interpreter, no garbage collector. the abstractions you wrote in—overloads, patterns, the railway—do not appear in the profile, because by the time tsuru is done folding, they aren't in the program anymore.

ten million frames

recursion is kanso's only loop, so the compiler doesn't treat tail calls as an optimization it might apply in a good mood—every call in tail position is a guaranteed tail call. mutual recursion, ten million frames deep, in constant stack:

pingpong.kso
fn even 0
  true

fn even n
  odd (n - 1)

main =
  verdict = even 10000000
  print "10000000 even? {verdict}"

fn odd 0
  false

fn odd n
  even (n - 1)
kanso build pingpong.kso --release && time ./pingpong
10000000 even? true
./pingpong  0.00s user 0.00s system 1% cpu 0.209 total

the guarantee is visible in the ir—musttail is llvm's strongest promise, an instruction the backend is required to honor:

pingpong.ll
  %t12 = musttail call tailcc %KValue @d_odd_1(%KValue %t11)
  ...
  %t12 = musttail call tailcc %KValue @d_even_1(%KValue %t11)

what the compiler actually does

no magic in any of this—just the front end proving things and llvm multiplying the proof. kanso build runs whole-program inference, monomorphizes every function per concrete instantiation (dispatch resolves statically—the "only switch" from chapter 03 costs nothing at runtime in monomorphic code), and emits llvm ir that clang compiles at -O3. patterns become field offsets. and a literal-dispatch group discriminating on one parameter—the scanner shape that kanso-json is made of—compiles to a genuine jump table. here is _is_number_char, whose arms are the byte literals 43 45 46 69 101 (+ - . E e), in the emitted ir:

jsonbench.ll
  switch i64 %t3, label %arm6 [
    i64 43, label %arm0
    i64 45, label %arm1
    i64 46, label %arm2
    i64 69, label %arm3
    i64 101, label %arm4
  ]

seven overloads in the source, one indexed branch in the machine. this is the book's closing argument, and it's the same argument every chapter has made: kanso deletes things—commas, formatters, wrappers, exceptions, mocks, and finally the runtime cost of its own abstractions. what's left, in the source and in the profile alike, is only the decisions.

tsuru

profile before you tune. the profile of a kanso binary names your functions, monomorphized—if the hot symbol is yours, the algorithm is the problem; if it's memmove, you are already at the metal and should go do something else with your afternoon.

mugi

her binary and my evaluation must agree byte for byte on every example, every ci run—including the whole json suite. when the compiler rewrites everything and not one output bit moves, you may trust the folding.
