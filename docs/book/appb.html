builtin reference

kanso keeps its standard library small enough to hold in your head. there are thirty builtins, and they come in two shapes. most are ordinary pure functions—sort takes a list and returns a sorted one, sqrt takes a number and returns its root, with no trace left behind. seven are descriptions: print, read_file, write_file, args, stdin, sleep, and random don't perform i/o when evaluated—they return a value describing i/o that the runtime performs later (chapter 05). this appendix marks those description and treats the rest as plain functions.

every builtin follows the same failure convention. an err or none handed to a builtin passes straight through, unlooked-at, so failures ride the railway from chapter 04 without special cases. a genuine type mismatch—sum on a list containing a string, sqrt on a record—is a program bug, not a value, and raises a runtime error at the point of the call. each entry below states both: what the builtin does, and what it does when the input is wrong. every example was run through the interpreter; every output block is copied from the machine, and the sources live under samples/appb/.

args at bytes char_code chars concat entries filter find2 from_code if join length map print push put random read_file round slice sleep sort sqrt stdin sum to_float to_int utf8 write_file

the sections that follow group the builtins by what they operate on—numbers, text, lists, maps, bytes, choosing, and effects—and end each group with one program that exercises the lot. within a group the entries are alphabetical; the index above is the fast path when you know the name.

numbers

round

round number → int

rounds a float to the nearest integer, ties away from zero (2.5 becomes 3). an int passes through unchanged, so round is safe to call on a value you already believe is whole. a non-number argument is a runtime error.

sqrt

sqrt number → float64

the square root, always a float—an int argument widens before the root is taken, so sqrt 9 is 3.0, not 3. a non-number argument is a runtime error.

to_float

to_float (string | int | float) → float64 | err

parses text into a float. an int widens and a float passes through, so the one function covers both parsing and numeric widening. text that isn't a number returns an err value carrying the offending string—a failure you can dispatch on, not a crash.

to_int

to_int (string | int) → int | err

parses text into an integer; an int passes through unchanged. unparseable text returns an err. because that err is a value, it rides the railway: interpolate it, or pipe it, and it propagates to the endpoint rather than being silently swallowed.

numbers.kso
root = sqrt 2
print "sqrt {root}"
  >> print "sqrt-int {sqrt 9}"
  >> print "round {round 2.6}"
  >> print "round-half {round 2.5}"
  >> print "to_int {to_int "42"}"
  >> print "to_float {to_float "3.14"}"
  >> print "to_float-int {to_float 5}"
kanso run numbers.kso
sqrt 1.4142135623730951
sqrt-int 3.0
round 3
round-half 3
to_int 42
to_float 3.14
to_float-int 5.0

the failure arm is worth seeing on its own. feed to_int a word and the err it returns is interpolated into the string, which propagates it—so the print never happens and the failure surfaces at the endpoint, its birthplace named:

parse_fail.kso
print "parsed {to_int "seven"}"
kanso run parse_fail.kso
error[endpoint]: unhandled err reached main: ""seven" is not an integer"
  born in main at parse_fail.kso:1

text and characters

char_code

char_code string → int

the unicode code point of a one-character string—char_code "A" is 65. the argument must be exactly one character; anything longer or shorter is a runtime error. its inverse is from_code.

chars

chars string → string[]

splits a string into its characters, each a one-character string, one per unicode scalar. accented letters count once—chars "café" has four elements, not five. a non-string argument is a runtime error.

from_code

from_code int → string | err

the one-character string for a code point—from_code 233 is "é". an integer that is not a unicode scalar value returns an err, so surrogate halves and out-of-range numbers fail as values rather than producing garbage.

join

join string[] string → string

joins a list of strings with a separator between them—the counterpart to splitting. a failure inside the list propagates; a non-string element is a runtime error.

length

length (string | list | map) → int

a count: characters of a string, elements of a list, entries of a map. one function serves all three because "how many" means the same thing across them. anything else is a runtime error.

slice

slice (string | list) int int → (string | list)

a sub-range by 1-based inclusive positions—slice "kanso" 2 4 is "ans", and the same call shape works on lists. out-of-range or inverted bounds yield an empty result rather than a failure, so slicing never surprises you with a none to handle.

strings.kso
cs = chars "café"
print "chars {cs}"
  >> print "length {length "café"}"
  >> print "char_code {char_code "A"}"
  >> print "from_code {from_code 233}"
  >> print "join {join ["a" "b" "c"] "-"}"
  >> print "slice {slice "kanso" 2 4}"
  >> print "concat {concat [1 2] [3 4]}"
kanso run strings.kso
chars ["c" "a" "f" "é"]
length 4
char_code 65
from_code é
join a-b-c
slice ans
concat [1 2 3 4]

lists

at

at (list | string | map) index → value | none

indexing. a list or string is indexed by a 1-based position; a map is indexed by its key. at is what the subscript form xs[i] desugars to. a missing position or absent key returns none—a dispatchable value, not an exception—so the caller decides what "not there" means. (an out-of-range subscript with xs[i] instead raises; at is the soft form.)

concat

concat list list → list

appends two lists into one. both arguments must be lists; the result is a new list and neither input is disturbed.

filter

filter list (value → bool) → list

keeps the elements for which the predicate returns true. the predicate must answer true or false; any other result is a runtime error, which catches a predicate that forgot to return a boolean.

map

map list (value → value) → list

applies a function to every element and collects the results in order. the function is any callable—a named function, a lambda, a builtin.

push

push list value → list

a new list with one value appended. it is functional: the original list is unchanged, and the value's type is unconstrained. under the hood the runtime reuses the buffer in place when the list is uniquely owned, so the value semantics cost nothing when nobody else is looking.

sort

sort list → list

ascending order. the elements must be mutually comparable—all ints, all strings—and a list mixing types is a runtime error, because there is no honest order across types. sorting is stable and returns a new list.

sum

sum list → int

adds up a list of integers, empty list summing to 0. a failure in the list propagates; a non-integer element is a runtime error.

lists.kso
xs = [3 1 2]
print "map {map xs (n -> n * 10)}"
  >> print "filter {filter xs (n -> n > 1)}"
  >> print "sort {sort xs}"
  >> print "sum {sum xs}"
  >> print "push {push xs 4}"
  >> print "at {at xs 2}"
  >> print "length {length xs}"
  >> print "slice {slice xs 1 2}"
  >> print "concat {concat xs [9]}"
kanso run lists.kso
map [30 10 20]
filter [3 2]
sort [1 2 3]
sum 6
push [3 1 2 4]
at 1
length 3
slice [3 1]
concat [3 1 2 9]

note at xs 2 is 1—position two of [3 1 2], one-based. when the position is absent, at hands back none, and because none is a value you dispatch on it. two overloads of describe—one that matches none, one that matches anything—turn "missing" into an ordinary branch:

missing.kso
fn describe none
  "index out of range"

fn describe x
  "value is {x}"

pub play =
  found = describe (at [10 20 30] 2)
  gone = describe (at [10 20 30] 9)
  print "in-range: {found}" >> print "missing: {gone}"
kanso play missing.kso
in-range: value is 20
missing: index out of range

the direct call describe (at …) is deliberate. the pipe . short-circuits a failure before it reaches the function, so at … . describe would propagate the none untouched; to dispatch on none you hand it to the function as an argument, where the none overload can match it.

maps

entries

entries map → entry[]

the key–value pairs of a map, each an entry key value record, returned in sorted key order. the order is deterministic, always—iteration over a map never depends on insertion history. to read a single value by key, use at.

put

put map key value → map

a new map with key set to value, replacing any previous binding. keys are ints or strings. like push it is functional and made in-place by the runtime when the map is uniquely owned. building a map from nothing starts at the empty-map literal [:]; a literal with known keys is written ["a": 1 "b": 2].

maps.kso
base = ["a": 1 "b": 2]
grown = put base "c" 3
print "literal {base}"
  >> print "put {grown}"
  >> print "at {at grown "b"}"
  >> print "entries {entries grown}"
  >> print "length {length grown}"
  >> print "empty {[:]}"
kanso run maps.kso
literal ["a": 1 "b": 2]
put ["a": 1 "b": 2 "c": 3]
at 2
entries [entry "a" 1 entry "b" 2 entry "c" 3]
length 3
empty [:]

bytes and encoding

bytes

bytes string → int[]

the utf-8 encoding of a string as a list of byte values (0–255). an ascii string gives one byte per character; "café" gives five bytes, because é takes two. this is the entry point to byte-level work—scanning, chunking, decoding—used heavily by the json library in chapter 08.

find2

find2 int[] int int int → int

scans a byte list from a 1-based start index for the first byte equal to either of two targets—hence the 2—and returns its 1-based position. when neither target is found it returns length + 1, one past the end, a sentinel you can compare against without a branch. it is the hot primitive behind tokenizers that stop at "the next comma or brace".

utf8

utf8 int[] → string | err

the inverse of bytes: decodes a list of byte values back into a string. a value outside 0–255, or a byte sequence that isn't valid utf-8, returns an err—decoding untrusted bytes fails as a value you can handle, never as a crash.

bytes.kso
bs = bytes "café"
print "bytes {bs}"
  >> print "utf8 {utf8 bs}"
  >> print "find2 {find2 (bytes "a,b;c") 1 44 59}"
kanso run bytes.kso
bytes [99 97 102 195 169]
utf8 café
find2 2

the find2 call searches the bytes of "a,b;c" from position 1 for a comma (44) or a semicolon (59). the comma sits at position 2, so that is the answer; had neither appeared, the result would have been 6—one past the five bytes.

choosing

if

if bool a b → value

chooses between two values by a boolean condition, and it is the only builtin that does not evaluate all its arguments: the untaken branch never runs, so if is safe to guard an expensive or failing computation. the condition must be true or false; anything else is a runtime error. in most code the dispatch of chapter 03 replaces if entirely—if is for the small, local two-way choice that isn't worth a second overload.

control.kso
fn classify n
  if (n > 0) "positive" "non-positive"

pub play = print "seven {classify 7}" >> print "zero {classify 0}"
kanso play control.kso
seven positive
zero non-positive

effects

the seven builtins in this group return descriptions, not results. evaluating them performs no i/o; they build a value the runtime executes afterward, which is why the same program can be run, planned with --plan, or fed to a scripted executor in a test (chapter 05). >> sequences two descriptions that pass no data; . pipes the value a description will yield into a function.

args description

args → string[]

yields the command-line arguments—everything after -- on the invocation—as a list of strings. because the list exists only at execution time, indexing past its end fails then, at the executor, on the same railway as any other err.

argv.kso
fn first xs
  xs[1]

pub play = args . first . print
kanso play argv.kso -- clay
clay
kanso play argv.kso
error[endpoint]: unhandled err reached the executor: "missing index 1"
  born in first at argv.kso:2

print description

print string → description

a description of writing one line to standard output. it takes a string, and only a string—to print a number or a list, interpolate it into one with "{value}". handing print a bare non-string is a runtime error with exactly that advice. sequence prints with >>; render the sequence without running it with --plan.

print.kso
print "first" >> print "second" >> print "third"
kanso run print.kso
first
second
third
kanso run print.kso --plan
plan:
  print "first"    # from line 1
  print "second"    # from line 1
  print "third"    # from line 1

random description

random int → int

a description that yields an integer in the half-open range [0, bound). the scheduler's generator is deterministic, so a program's rolls are reproducible run to run—random enough for a simulation, repeatable enough for a test. pipe the yielded value into a function with ..

sleep description

sleep int → description

a description of pausing for a number of milliseconds. under the deterministic scheduler it advances virtual time rather than wall-clock time, so a program that "sleeps" is still instant to test. sequence it with >>.

timing.kso
pub play = sleep 10 >> random 6 . show

fn show n
  print "rolled {n}"
kanso play timing.kso
rolled 2
kanso play timing.kso --plan
plan:
  sleep 10
  random 6
  . <continuation>

the plan names sleep and random up front, then stops at . <continuation>—honest notation for "what show does depends on a value the roll hasn't produced yet." that is the visible seam between >>, which lists every step, and ., which threads a not-yet-known value forward.

read_file description

read_file string → string

a description that yields the contents of a file as a string. a missing or unreadable path fails at execution time—at the executor, when the read is actually attempted—rather than when the description is built. pipe the contents into a function with ..

write_file description

write_file string string → description

a description of writing a string to a path, replacing any existing contents. it yields nothing useful, so it is normally sequenced with >> before the next step. here one program writes a file and then reads it straight back:

files.kso
write_file "note.txt" "one taiyaki" >> read_file "note.txt" . print
kanso run files.kso
one taiyaki

stdin description

stdin → string

a description that yields all of standard input as a single string. pipe it into a function to process what was fed on the pipe:

input.kso
pub play = stdin . shout . print

fn shout text
  "{text}!!"
printf 'hello from a pipe' | kanso play input.kso
hello from a pipe!!
err

hand me to a builtin and i pass straight through—sort, map, sum, none of them inspect me, they just return me. a type mistake is different: sum on a list with a string in it is a bug in your program, and that raises where it happens. failures flow; bugs stop.

that is the whole surface. thirty names, two shapes, one failure convention. you now have the signature, the behavior, and the failure mode of every builtin kanso offers—enough to read any program in this book without guessing what a call does, and enough to reach for the right primitive instead of hand-rolling it. the chapters build real programs on exactly these thirty; the json library of chapter 08 is little more than bytes, find2, and slice worn smooth.

exercises

  1. write a program that reads a file of numbers, one per line, and prints their sum. you will need read_file, chars or a split, to_int, and sum—decide how a non-numeric line should behave, and make that behavior visible in the output.
  2. using only bytes, find2, and slice, split a string on its first comma into a "before" and an "after". confirm with --plan that no i/o happens until you print the two halves.
  3. at returns none for a missing index while the subscript xs[i] raises an err. write one small program that shows both on the same out-of-range access, and explain in a comment which you would choose for a lookup that is expected to miss.
  4. write a two-overload label function—one arm matching none, one matching a value—and drive it from a random roll that indexes into a short list. run it a few times and note that the roll is the same each run.
