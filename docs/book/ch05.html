chapter 05

effects are descriptions

time to confess what print has been doing all book: nothing. every kanso function is pure—evaluating print "hi" performs no i/o. it returns a description of printing, a value like any other. your entire program is one big pure computation whose final answer is main: a description of everything the program intends to do. the runtime receives that description and walks it. evaluation decides; the runtime performs.

you can see the seam directly, because a description is data and data can be rendered instead of executed. >> glues descriptions into bigger descriptions—"this, then that"—and --plan shows the glued-up value without running it:

brew.kso
main = print "brewing" >> print "steeping" >> print "pouring"
kanso run brew.kso --plan
plan:
  print "brewing"    # from line 1
  print "steeping"    # from line 1
  print "pouring"    # from line 1
kanso run brew.kso
brewing
steeping
pouring

--plan isn't a debugger bolted on afterwards; it falls out of the design. the plan already exists as a value—the flag just prints it instead of handing it to the executor. each step carries the source span it came from, which is where the provenance comments come from.

pipes, and pipes into effects

the other operator you've been glimpsing is .—the pipe. x . f is f x, read left to right, so data flows the way your eye does:

pipes.kso
fn greet name
  "hello, {name}"

main = "kanso" . greet . print
output
hello, kanso

now the interesting move: pipe out of a description. read_file path doesn't return a string—it returns a description of reading, and the file's contents don't exist until the runtime performs it. so when you pipe a description into a function, kanso wires the function to receive the value once it exists. piping into a description is bind:

flavors.txt
matcha
ube
hojicha
count.kso
fn count_lines text
  newlines = filter (chars text) (c -> c == "\n")
  "flavors.txt has {length newlines} lines"

main = read_file "flavors.txt" . count_lines . print
kanso run count.kso
flavors.txt has 3 lines
kanso run count.kso --plan
plan:
  read_file "flavors.txt"
  . <continuation>
  . <continuation>

look at that plan. the runtime can name the first step, but the rest is <continuation>—honest notation for "what happens next depends on a value i don't have yet." that's the difference between >> and . in one screenshot: >> sequences descriptions that don't pass data (the plan lists every step up front), while . threads a value through, so later steps are functions waiting for input. together they are the complete vocabulary of "before" in kanso—everywhere else, order belongs to the runtime.

note what count_lines is: an ordinary pure function that takes a string. it doesn't know files exist. purity isn't quarantined into special syntax—there's no async, no coloring, no effect annotations on your functions. effect sets are inferred and ride up the call graph on their own.

the rest of the i/o vocabulary

writing is a description too—sequence it with >> like anything else:

save.kso
main =
  order = "one taiyaki, extra custard"
  write_file "order.txt" order >> print "order saved"
kanso run save.kso
order saved
cat order.txt
one taiyaki, extra custard

args is the argument list (everything after -- on the command line), and stdin is standard input, both descriptions, both piping like everything else:

welcome.kso
fn first xs
  xs[1]

fn greet name
  "irasshaimase, {name}"

main = args . first . greet . print
kanso run welcome.kso -- clay
irasshaimase, clay
kanso run welcome.kso
error[endpoint]: unhandled err reached the executor: "missing index 1"

run it with no argument and the strict index inside first fails—at execution time, because that's when the argument list exists. the err rides the same railway as chapter 04, and the endpoint rule catches it at the executor instead of main. failure during effects isn't a special kind of failure; it's the same value on the same rails.

shout.kso
main = stdin . shout . print

fn shout text
  "{text}!!"
printf 'hello from a pipe' | kanso run shout.kso
hello from a pipe!!

testing without mocks

here is what this architecture buys, and it's the reason the whole design exists. in most languages, testing effectful code means interception: monkey-patch the i/o call, record what would have happened, hope the patch is faithful to the real thing. in kanso there is nothing to intercept, because evaluation never does i/o. a test evaluates main—pure, fast, no side effects—and gets the program's entire intent as a value.

the interpreter makes this concrete. its runtime is a trait with two implementations—the real executor performs effects, and a scripted one appends each effect to a transcript:

src/eval.rs
pub struct ScriptedExecutor {
    pub transcript: Vec<String>,
    pub script_args: Vec<String>,
    pub script_stdin: String,
    pub files: std::collections::HashMap<String, String>,
}

a test hands main's description to the scripted executor—feeding it canned args, canned stdin, an in-memory map of files—and then asserts on the transcript with plain data equality: assert_eq!(executor.transcript, ["print \"a\"", "print \"b\""]). no stubbing framework, no argument matchers, no "verify this was called once." the program said what it would do, and the test reads the saying.

mugi

when a program surprises you, reach for --plan before you reach for a debugger. the plan is what your program is; running it is merely what it does. most bugs are visible in the intent.

err

i work execution time too. a file that isn't there, an argument that wasn't passed—same me, same railway, caught at the executor instead of main. you do not need a second error model for i/o, and kanso declines to sell you one.
