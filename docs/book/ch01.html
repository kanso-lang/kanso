chapter 01

hello, kanso

every language book opens the same way, and there is a reason: until something runs on your machine, everything else is theory. so — clone, build, run. the toolchain is one binary and it is built with cargo.

shell
$ git clone https://github.com/ClayShentrup/kanso
$ cd kanso
$ cargo build --release
$ ./target/release/kanso run examples/hello.kso
hello, world

put target/release on your path (or alias it — we'll write plain kanso from here on) and you have the whole toolchain. there is no package manager to configure, no project scaffold to generate, no manifest file asking you questions. a kanso program is a file.

the whole program

here is hello world in its entirety:

hello.kso
main = print "hello, world"
kanso run hello.kso
hello, world

one line, and it's a slightly subversive line. main is not a function — it's a constant, bound with plain =, and its value is a description of printing. nothing executes while main is evaluated; the runtime receives that description and performs it. this distinction will carry an entire chapter later. for now, notice only that the simplest possible program is a value with a name.

notice also what isn't there. no parentheses around the argument — application is a space, print "hello, world". no semicolon. no import for print. and no room for stylistic debate: this program has exactly one rendering, and you just read it.

four verbs

the binary answers to four commands, and together they are the whole development story.

kanso run interprets a file directly — no codegen, no artifacts, the full pipeline cold in about ten milliseconds. this is the dev loop, and you will live in it. kanso check runs the same pipeline but stops before executing; it exists so editors and ci can ask "is this program legal?" and get a one-line answer:

shell
$ kanso check hello.kso
hello.kso: ok

kanso test runs tests, which in kanso are just constants whose names begin with test_ — a test is a boolean value, and the runner evaluates it. no framework, no assertion library, no dsl to learn:

greet_test.kso
fn greet name
  "hello, {name}"

test_greet = (greet "kanso") == "hello, kanso"
kanso test greet_test.kso
test_greet ... ok
1 passed, 0 failed

and kanso build is the other engine entirely: it lowers your program to llvm ir and hands it to clang for a native binary. the default build is tuned for the loop — your program compiles unoptimized against a cached, already-optimized runtime, so warm builds land in tens of milliseconds. --release is the full-fat version: whole-program lto, the slowest build, the fastest binary, and the mode every number in chapter 07 was measured under.

shell
$ kanso build hello.kso
built ./hello (llvm ir at hello.ll)
$ ./hello
hello, world

two engines, one meaning

most toolchains make you pick a side. go bought instant compiles and pays with slower binaries forever; rust bought maximum binaries and makes every developer wait. kanso declines the bundle by shipping both engines: the interpreter is the dev loop and never touches a backend, the compiler is the release path and never compromises. the price of having two is that they must agree — so they are forced to. ci builds every example native and requires byte-identical output against the interpreter. a one-bit disagreement fails the build.

you will meet both engines properly in chapter 07, where the compiler gets to show off. until then, everything in this book runs under kanso run, and everything would produce the same bytes under kanso build. that's not a slogan; it's a test suite.

mugi

kanso run is me. i keep nothing between runs — no cache, no build directory, no lockfile. if your edit-run loop ever feels slower than your thoughts, file a bug; i consider it a personal failure.

tsuru

whatever mugi says your program means, my binary must say byte for byte, or neither of us ships. reach for kanso build when you want a release artifact; reach for him the other thousand times a day.
