chapter 04

failure is a value

kanso has no exceptions and no panics. it also has no Ok, no Some, no .unwrap()—none of the wrapper ceremony that result types usually charge. success is the bare value. failure is err reason, an ordinary value of an ordinary type. absence is none, a different ordinary value. that's the entire error model, and this chapter is about how far it goes.

start by breaking something:

ratio.kso
main =
  ratio = 10 / 0
  print "ratio: {ratio}"
kanso run ratio.kso
error[endpoint]: unhandled err reached main: "division by zero"

read that carefully, because it is not a crash. division by zero produced an err value, the value flowed to main, and the endpoint rule fired: an err reaching the top of the program with no arm to receive it is an error in its own right. nothing was thrown; a value arrived somewhere that had no plan for it, and kanso considers having no plan the bug.

an arm for the err

the plan is dispatch—the only switch, doing its second job. err reason is a constructor pattern like any other, so handling failure is just one more overload:

handle.kso
fn describe (err reason)
  "failed: {reason}"

fn describe n
  "got {n}"

main =
  ratio = 10 / 0
  print (describe ratio)
output
failed: division by zero

no try, no catch, no rescue block changing the shape of the code. the happy path and the sad path are two arms of the same function, sitting at the same indentation, both just returning strings.

the railway

here is the part that deletes real code from real programs. when an err flows into a function with no arm for it, the function doesn't run—the err passes through, untouched, automatically. watch a failure born three calls deep ride past every function on the way up:

railway.kso
fn describe (err reason)
  "could not price the order: {reason}"

fn describe total
  "order total: {total}"

main =
  good = total_price 120 4
  bad = total_price 120 0
  print (describe good) >> print (describe bad)

fn total_price cents people
  share = cents / people
  with_tip share

fn with_tip share
  share + (share / 10)
output
order total: 33
could not price the order: division by zero

count the lines of error handling in total_price and with_tip: zero. the division by zero happens inside total_price, the resulting err flows into with_tip—whose arms don't match it, so it flows through—back out to main, and finally into describe, the first function that declared an arm for it. this is the railway: the sad track runs parallel to the happy one, and only stations with an err platform can receive the train. functions in the middle state their happy-path logic and nothing else, ever.

none is not err

absence gets its own value, and the standard library is opinionated about which one you should meet. indexing with xs[i] is strict: it means "i know this element exists," and a miss is an err—a broken promise:

ninth.kso
main =
  flavors = ["matcha" "ube" "hojicha"]
  print "ninth: {flavors[9]}"
kanso run ninth.kso
error[endpoint]: unhandled err reached main: "missing index 9"

(indexing is 1-based, everywhere, no exceptions. the first flavor is flavors[1]. we are at peace with this.)

when absence is expected—you're asking, not promising—use at, which returns none on a miss. and none, like err, is dispatched on, not checked for:

fourth.kso
fn describe none
  "no fourth flavor"

fn describe x
  "fourth flavor: {x}"

main =
  flavors = ["matcha" "ube" "hojicha"]
  first = flavors[1]
  fourth = at flavors 4
  print "first: {first}" >> print (describe fourth)
output
first: matcha
no fourth flavor

the same pair of tools covers maps—literal syntax ["dango": 350], strict lookup with brackets, polite lookup with at:

menu.kso
fn describe none
  "not on the menu"

fn describe price
  "{price} yen"

main =
  menu = ["dango": 350 "taiyaki": 500]
  print (describe (at menu "pocky"))
output
not on the menu

choosing between them is a design act, and the language makes you commit: brackets document an invariant ("this key exists, and if it doesn't, that's a bug—fail here"), at documents a question ("this may be absent, and absence is a case my code handles"). the distinction other languages leave to comments and code review, kanso puts in the call site.

err

people confuse me with my cousin none. none is what you get when you asked—a lookup that may miss. i am what you get when something broke. seat us at different tables and your program's intent reads itself.

err

i never sneak. i knock at every function on the way up, and the endpoint rule guarantees somebody eventually opens the door—because pretending i'm not there refuses to compile. write the arm where the context to respond actually lives, and nowhere else.
