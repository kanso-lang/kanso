the compiler

the semantics are the license

every language pitches its restrictions as virtue. kanso's restrictions are also collateral. every constraint the language imposes for taste—functions are pure, programs have one canonical rendering, dispatch is the only branch, nothing mutates—is exactly the precondition some aggressive compiler technique needs to be sound. techniques that are heroic engineering elsewhere (speculation with deoptimization, fusion, whole-program specialization) stop being heroic here, because the semantics already promised what their proofs require. the optimizer doesn't work around the language; it cashes it in.

this page is the running ledger of that trade: what the license has already paid for, with receipts, and what it's queued to pay for next. rulings land at the bottom as queue items and graduate up into sections here—this is a living document, and it only grows.

01three engines, one oracle

kanso's toolchain is three independent implementations of one semantics: a tree-walking interpreter written to be obviously correct, an llvm-backed native compiler written to be fast, and a direct-to-wasm backend written to fit in a browser tab. ci holds all of them to byte-identical output across the entire golden corpus—every example, every diagnostic, every endpoint message. the guarantee is about rendered corpus bytes, floats included, because every engine renders the shortest round-trip that survives re-parsing—not about bit-level float arithmetic in arbitrary programs. the architecture and its economics are told in about, part 01; what belongs here is why a compiler engineer should care: a compiler tested only against itself moves confidently in whatever direction its bugs point. a compiler tested against a naive oracle can't.

the differential lattice has caught real bugs, each one named to the exact case by a one-bit disagreement: a float-rendering divergence (C's %.17g against rust's shortest round-trip—the runtime now searches for the shortest rendering that round-trips), a utf-8 validation gap, and—the day kanso run went native—an interpolation-propagation bug (templates stringified failures instead of propagating them) and a missing pattern form in the backend itself. none of these needed a bug report. they needed a diff.

02dispatch becomes a jump table

kanso has no if on values in the branching sense other languages mean—dispatch is the only switch. that's a taste ruling, and it hands the compiler something most front ends never get: every branch in the program is a structured, enumerable decision over declared alternatives, not an opaque boolean the optimizer must reverse-engineer. a literal-dispatch overload group discriminating on one parameter—the scanner shape kanso-json is made of—compiles to a genuine llvm switch, which the backend lowers to an indexed jump table. seven overloads in the source, one indexed branch in the machine. the receipts, ir included, are in chapter 07.

03tail calls are a guarantee, not a mood

recursion is kanso's only loop, so tail-call elimination cannot be an optimization the compiler applies when it's feeling lucky—it's part of the language's contract. every call in tail position is a guaranteed tail call: the native backend emits musttail, llvm's strongest promise, an instruction the code generator is required to honor; the browser backend emits wasm's return_call. the receipts: ten million frames of mutual recursion in constant stack natively, one million frames in constant stack inside a browser tab. deep recursion isn't a stylistic risk you manage with a stack limit; it's just how iteration is spelled.

04inference deletes the guards

a dynamic-feeling language usually pays a runtime tax: tag checks before every operation, failure guards after every call. kanso's whole-program inference—the monotone fixpoint whose near-death story is about, part 03—computes, per call site, the exact set of concrete type vectors that can arrive. the emitter then monomorphizes per vector: dispatch resolves statically, patterns become field offsets, and on proven paths the tag checks and failure guards are simply deleted—not predicted well, not branch-hinted, absent.

the receipt is the benchmark: a 188kb nested json document, read at runtime and decoded 150 times, averages ~1.8ms per decode through the llvm backend—ahead of go's encoding/json at ~1.9ms, from a pure language with automatic failure propagation and no mutation. but the honest measuring stick is serde_json, the parser a rust team actually ships: it does the same work in ~0.86ms, about twice as fast. that gap is not a mystery—the cpu profile of the kanso binary shows memmove and map insertion on top, because building an n-key object still copies the whole map on every insert. it is the reference counting that has not shipped, not the abstractions, that serde is beating. full numbers and the reproduction recipe live in bench/RESULTS.md; §09 is how the gap closes.

05run is build-and-run

kanso run does not interpret your program; it compiles it and runs the binary, and makes that indistinguishable from interpretation by caching. the dev binary is keyed by a content hash of the generated ir: an unchanged program re-runs in about five milliseconds with no compiler invocation at all—faster than the interpreter could even start—and a changed one rebuilds in tens of milliseconds, unoptimized, against a cached, already-optimized runtime object. -O3 -flto exists only where it belongs, behind --release. the trade every toolchain gets bullied into—instant compiles or fast binaries, pick one—dissolves when the optimizer is never asked to be the dev loop. the interpreter stays on as the oracle and the interactive tier (--plan, the repl, the playground), which is exactly the shape section 01 requires anyway.

06arbitrary precision without the bill

int is arbitrary precision. that is semantics—it holds on every engine, and it is not for sale to the optimizer. what is the optimizer's to decide is how rarely you pay for it: the compiler emits each function in tiered versions. the fast tier compiles speculatively against raw int64—byte-for-byte the fast codegen—with overflow checks that bail out of the version, restarting the call in a second compilation of the same source against heap bignums. the bailout is an internal signal, never a language-level value.

notice what makes that sound. restarting a half-executed function is a catastrophe in any language where the function might have already done something—printed, mutated, sent. kanso functions are pure and effects are descriptions, so a restart is unobservable by construction. this is the javascript engine's best trick—speculate on the probable type, deoptimize on surprise—performed ahead of time, with zero deoptimization machinery: no on-stack replacement, no deopt metadata, no interpreter to fall back into. just run the other version. the speculative-compilation apparatus jit engines bleed for falls out of the semantics. and the mechanism was designed as a general one from day one: int64-not-bignum is merely the first probable-but-unproven fact worth betting on.

07the data rulings

three rulings about data shape landed together, and each one hands the compiler more structure while handing the human less to remember.

destructures always wear braces. { artist minutes } = song binds by field name, subset or full; { artist: who } = song renames; artist = song is always and only an ordinary binding. an earlier draft elided braces for multi-name reads and died in review: arity-dependent meaning is a refactoring landmine—deleting one name from a bare destructure would have silently turned a field read into a whole-record alias. the positional binding form died with it, because positional binds reshuffle when a type grows a field. one construct, one look, at every arity—and for the compiler, every destructure is field-name resolution against a known record type, which compiles to field offsets and nothing else.

a type may declare zero fields. type null—one line, no body. its bare mention is its value; in pattern position the bare name matches the marker, unambiguous because no-shadowing already forbids a binding from wearing a declared type's name. it behaves like true/false/none mechanically but is ordinary data: null is a library declaration, never a keyword, and json's null sheds the dummy field it once carried. markers weren't a new concept; they were record types with the field count at zero.

a field may list several permitted types. title: null string—space-separated, alphabetized like every enumeration in the language. reads dispatch on what the value is; in clay's sentence, verbatim: "you're never able to use a conditional if to evaluate whether it's a value or null; you just have a polymorphic call." go's (value, ok bool) pattern mandates a forgettable conditional at every consumption site; typeset fields make the arms checkable and the conditional unwritable. constructors enforce membership—a compile error where provable, an endpoint-reported defect otherwise. and several markers in one field is enums for free (status: active archived pending): zero new machinery, because enums weren't a new concept either. for the compiler, a typeset field is a closed tagged union it can enumerate, specialize, and—under inference—often erase.

08the backend in the tab

the third engine emits wasm bytecode directly—hand-rolled section encoding, no llvm within a hundred miles, because llvm is a dependency measured in hundreds of megabytes and a browser tab is not. its value representation is deliberately strange: every kanso value is an i32 handle into a registry owned by the toolchain module, which reuses the interpreter's value machinery so semantics stay oracle-identical. the emitted program module owns no linear memory at all—literals are pre-registered at compile time, so the module is pure control flow over handles. host-driven iteration (map and filter walking a list, description continuations) calls back into compiled closures through a single imported trampoline, k_callback, wired by the javascript glue to the program's function table.

this is how the playground compiles the program you type into machine code inside the tab, tail calls included, with the whole toolchain at about half a megabyte of wasm. the registry design is the first cut, not the final one—the path to a self-contained module (its own memory, its own bignums, call_indirect retiring the trampoline) is scoped in design/wasm-runtime.md, and every step of that migration is verifiable because the differential harness pins the corpus to native-engine bytes.

09the frontier

the standing directive for what lands next: core changes are on the table iff they don't tax the user and the payoff is game-changing. the goal, stated plainly—faster than rust, with no ownership annotations and no rejected programs. the queue, each entry named with the semantic property that licenses it (the full scoring is in design/performance-frontier.md):

perceus reference counting with reuse analysis—queued next, and the main unplayed card for closing the ~2x gap to serde. compiler-inserted precise refcounting where refcount-1 values are updated in place: persistent-looking code at imperative speed. the prior art is named plainly—koka invented perceus, and lean 4 and roc ship it in production; kanso's edge is not being first but that no-shadowing and nothing-wasted make liveness exact, so the same analyses land more completely. the license: no mutation and tree-shaped values mean no cycles, so there is no cycle collector and none of the usual caveats. the residual gap, stated up front: where uniqueness can't be proven, a reuse site pays one predictable branch—closeable only as an opt-in per-function fip guarantee, never globally, because closing it globally rebuilds the borrow checker. targets: map inserts (o(n) copies today), number building, every builder loop.

generalized aot speculation—section 06's mechanism, past ints: speculate on any probable-but-unproven inference fact (monomorphic dispatch target, list-not-map, ascii-not-utf8) with bail-and-restart. jit-engine speculation, ahead of time, zero deoptimization metadata.

cost-bound inference (the raml lineage)—type-system-driven polynomial cost bounds, solved by linear programming. pure code with structural recursion is exactly the tractable fragment, so kanso check can report asymptotic bounds and flag regressions—"decode was o(n), your change made it o(n²)"—and the same bounds feed the parallelism cost model. no mainstream-feeling language ships this.

equality saturation—rewrite by e-graph instead of ordered passes, applied to kanso's own ir before llvm. pipeline fusion (map f . map g becoming one pass) is unconditionally sound under purity: deforestation without haskell's fragility, and phase-ordering dissolves as a problem category.

simd structural scanning—simdjson-class byte classification feeding the jump tables the backend already emits. lives behind bytes primitives, no language surface; the remaining order of magnitude on parsing workloads.

interaction nets—the hvm/bend line stays on the watch list: spectacular on some workloads, unproven for strict everyday code, and adopting it is a runtime replacement rather than a pass. revisit when the strict-mode story matures.

the faster-than-rust thesis, stated honestly: not "beat rust at microbenchmark loops"—llvm emits the same instructions for both. the winnable claim is beating idiomatic rust on allocation-heavy real workloads, because perceus reuse eliminates the defensive clones the borrow checker pushes people into, fusion is unconditional under purity, speculation specializes what monomorphization can't see, and the cost model schedules parallelism the user never wrote. rust's performance requires the user to be excellent; kanso's should require the compiler to be.

10the memory manager that isn't

§09 queues perceus and states the thesis as no rejected programs. a night of design pushed straight through that promise and out the far side, and the exit is worth writing down even though none of it is built, because it moves the target. §09 asks how to make reference counting cheap. the question underneath is why there is a reference counter at all. for a closed-world, pure, cycle-free language the honest answer may be: there doesn't have to be one. not a leaner garbage collector, not a cheaper count—none. no collector, no counts, not even the header a count rides on. the compiler decides every object's fate—reuse in place, free here, copy there—entirely at compile time, and a program whose fate it cannot settle does not get a runtime fallback. it gets a compile error and a one-word fix.

the immediate objection is a theorem. uniqueness is a non-trivial semantic property, so by rice's theorem it is undecidable in general—no terminating analysis proves it for every program. and the usual consolation, the analysis misses cases a human can see, dissolves under a second look. a human "seeing" that a value is unique is producing a proof, and under the church–turing thesis that faculty isn't hypercomputational: where no proof exists, no one sees it, human included. uniqueness can genuinely reduce to whether some turing machine halts, and there are explicit machines whose halting is independent of set theory—yedidia and aaronson concretely built one. for such a program uniqueness is true but unprovable, and unprovable means invisible to everyone.

the way out is not a cleverer analysis chasing an undecidable property—that is a treadmill against rice you cannot win. it is to make the property decidable by construction: constrain the language so uniqueness is structural, so that "is unique" and "the checker accepts it" coincide for every program the language can express. then the analysis is not merely sound but complete, and rejection stops being a limitation and becomes a definition. inside such a language the philosophy claim is a theorem: if the compiler cannot prove a value unique, the value genuinely is shared, and a copy genuinely is needed—no rejection is ever spurious, and there is no case a human sits over knowing better, because the language cannot express the case where the human would be right. you design the halting problem out of the uniqueness question rather than losing to it. a linear or uniqueness type system already does this crudely—clean's uniqueness types, linear haskell (bernardy et al., POPL 2018), the quantitative type theory of atkey (LICS 2018)—by making usage something the type checker counts. the open move is doing it inferred and ergonomic instead of annotated and rigid.

inference is where closed-world stops being a footnote and becomes the whole edge. rust's borrow checker is local because it must survive separate compilation and unknown callers—it cannot see enough to infer, so it makes you write the proof as lifetimes. koka's perceus does the honest thing a higher-order language with separate compilation is forced into: it gives up proving uniqueness statically and tracks it at runtime, in the reference count, so reuse is opportunistic—a branch on the count at the moment of update (reinking, xie, de moura, leijen, PLDI 2021). koka's fip keyword (lorenzen, leijen, swierstra, ICFP 2023) claws back a static guarantee—it verifies a function runs allocation-free in constant stack—but only conditionally, "as long as the arguments are unique," and in general that argument-uniqueness still bottoms out in the runtime count. the static discipline stands on a dynamic substrate. kanso is closed-world: the compiler sees every call site of every function, no separate compilation, no unknown caller. whole-program uniqueness inference therefore has, in principle, the information rust's checker lacks and koka defers to runtime. what those systems can only annotate or count, a closed-world analysis can try to prove.

if it proves, the runtime disappears. there is no count to maintain, so there is no per-object header to carry it—every heap object becomes exactly its data, smaller and cache-friendlier. freeing is a drop the compiler inserts at the statically-known last use; reuse is the in-place mutation it emits where it proved the value unique; both are settled before the program runs. the reach of that: the compiler writes correct malloc, free, and in-place reuse for you, at rust's zero runtime overhead and with none of rust's annotations. not "as fast as rust with a gc-shaped asterisk"—no runtime memory management at all. that is the literal reading of faster-than-rust-without-lifetimes, and it is a stronger claim than §09's, which kept perceus and its count.

a reference-counted opt-in—koka's fip read in a mirror, static by default and rc where you asked out loud—is the obvious concession here, and it is a concession worth designing away. the cases that force a count are one shape: pointer-based dynamic sharing, a value reachable from two places whose lifetimes can't be statically ordered, so no one knows who frees it last. a count answers "who's last?" at runtime; the construct that never asks the question shares a value instead of a pointer. an explicit region owns the data and you share a plain index into it—copyable, ownerless, alias-free, and the region's lifetime is a static scope. dynamic sharing expressed with no dynamic-lifetime pointers stays fully provable, with no count, because no last-reader was ever in doubt. this is a real lineage, not a wish—tofte–talpin region inference (mlkit, cyclone), and the data-oriented "handles, not pointers" discipline rust game-dev and zig reach for precisely to escape the borrow checker—made the default rather than the escape hatch. and the honest cost lands here, not on reference counting: the safe form of index-sharing needs either a per-access generation check (vale's generational references, rc's cost in a different coat) or region discipline—freeing the pool as a unit so staleness can't arise—which trades prompt per-object freeing of shared data for total static provability. for tree-shaped work regions scope to a call or a request and the trade is nearly free; for a long-lived process juggling transient shared state it can bloat. reference counting, in this design, is the pocket card you deploy only if the constructs prove insufficient—and the aim is never to unpocket it. that is the sharpest split from koka, which keeps rc as the substrate: kanso would try to make the count expressible-away, not merely cheap.

the honesty tiers matter more here than anywhere else on this page, because none of it is built. proven, by others: precise reference counting with reuse and garbage-freedom for cycle-free programs (perceus); a checked fully-in-place discipline (fip); and fully-static memory management by whole-program analysis, which is old and real—tofte–talpin region inference and mlkit, and proust's ASAP thesis, which frees as early as it can prove. kanso's, and unproven: the specific stack—closed-world uniqueness inference with no annotations, a prove-or-copy totality that rejects rather than falls back, no runtime substrate at all, and rc demoted to a visible opt-in. every brick exists; the building does not, and "the pieces compose" is a claim, not a result.

the open questions are the ones that decide whether this is a language or a paper. first: can the inference be complete over a fragment that is actually pleasant to write—few copy points in real code? that is empirical, and the self-hosting compiler is the rig that answers it; a copy every third function means the constraint is too sharp and perceus-as-floor wins after all. second: the ergonomics-versus-predictability tension is real—the more the analysis infers, the harder it is for a reader to predict why a program was rejected, which is the borrow checker's own failure mode returning through the optimizer. third, the honest hard case: fine-grained persistent sharing—the clojure-style immutable structure whose subtrees are shared every direction—is exactly where "no runtime count" bites, because promptly freeing a shared subtree is what a count is for; without one you over-retain or you copy. whether kanso's domain leans on that pattern is, once more, a question only real programs answer.

so the pivot is a trade, and §09's promise is what's traded. no rejected programs buys expressiveness with a runtime count you cannot see; prove-or-copy buys a cost model you can read straight off the source, at the price of a compiler that can say no. the no is lighter than rust's—a copy at a rare point, not lifetimes threaded through every signature—but it is a no, and pretending otherwise would fail this page's own honesty rule. which of the two is right for kanso is not knowable from a whiteboard. it is knowable from writing the compiler in the language and counting the copies. that is the next real experiment, and it is worth more than any benchmark on this page.

codathe mushroom test

every proposal on this page passed the same two filters before it earned a section. the first is the house rule you've met everywhere else on this site: polymorphism over conditionals—if the design answer is "the user writes an if," the design is wrong, because dispatch makes the arms checkable and the conditional unwritable. the second is the mushroom test. a mushroom looks like a new organism; dig, and it's a fruiting body of a mycelium that was already there. that's the question every ruling faces: does this proposal add a concept, or does it reveal that an existing concept already covers it? zero-field types looked like a feature and turned out to be record types with nothing left to remove. enums looked like a feature and turned out to be typesets of markers. the int tiers looked like machinery and turned out to be one instance of speculation-under-purity. proposals that add a concept wait; proposals that reveal one land—and then they graduate off the queue below and into the ledger above.

  • pub visibility—_name as module-private has two real defects: promoting a name renames every call site, and _ already means wildcard and deliberately-unused. the pub-modifier draft (private by default, every arm of a public group marked, api surface greppable in one pass) is written in design/visibility.md; awaiting the gavel.
  • parameter typesets—fields ship; parameters wait on the annotation-redundancy checker, so a guard the body already derives can be rejected as clutter before parameter guards become legal.
  • dispatch positional-field fragility—the destructuring ruling deleted positional binds because a type growing a field reshuffles them; dispatch arms still consume fields positionally, which is the same flank unguarded. wants litigation.
  • kq repo publish—committed and ready at ~/dev/kq with a readme and the jq benchmark; publishing awaits an explicit go.
  • kanso test --native—tests still interpret; heavy suites are exactly where the build-and-run crossover bites.
  • >> statically effect-only—runtime-enforced today; wants the effect-inference pass.
  • imports slice b—slash qualification, the prerequisite for retiring vendored copies of kanso-json.

appendix—field report

the json gauntlet

this was the site's first receipts page, kept verbatim as the record of the calls that built kanso-json; the rulings it provoked now live in the essay above.

the fastest way to find out whether a language is real is to make it earn a living. so we ported the job description of go's most-used package—encoding/json—into kanso: a complete decoder and encoder, escape handling including \uXXXX, failure positions, canonical output. it exists, it passes sixteen tests under kanso test, and every judgment call made along the way is recorded here in three honesty tiers: things the exercise proved, executive calls awaiting ratification, and friction we refuse to pretend we didn't feel.

tier 01what the gauntlet proved

dispatch is a parser's native language. a recursive-descent parser is one long "what character am i looking at?"—and kanso's literal-dispatch overloads are that question. the tokenizer's decision tables read like tables:

lib/json/json.kso
fn value_for "\"" cs p
  parse_string cs (p + 1)

fn value_for "[" cs p
  parse_array cs (p + 1)

fn value_for "t" cs p
  word cs p "true" true

fn value_for none _ p
  fail p "unexpected end of input"

auto-propagation deleted the error plumbing. all of it. the parser contains not one line of "check if the last step failed." a failure born anywhere—bad escape, invalid number, truncated input—rides the return values past every continuation function (whose constructor-pattern arms simply don't match it) and surfaces from decode with its position intact. the happy path is the only path anyone wrote. this was the design's biggest bet, and it paid in full.

nothing-wasted caught real noise. the compiler rejected seventeen dispatch arms for naming parameters they never used, forcing _ discards that now document, in the signature, exactly what each arm consumes. annoying for ninety seconds; correct forever.

end-of-input is not a special case. at past the end returns none, which propagates like any failure and gets caught by explicit none arms exactly where the grammar cares. eof handling cost zero new concepts.

tier 02executive calls—ratify or reverse

each of these is implemented, tested, and reversible. defaults chosen by the house rule: the right thing, by default, with nothing to configure.

1. kanso test. a test is a constant named test_* whose value is true. no framework, no assertion dsl—== on values is the assertion, because structural equality is already total. kanso run requires main; check and test don't (a library is valid kanso without an entry point).

2. map literals. ["a": 1 "b": 2], empty map [:]. keys are literals only (dynamic maps are built with put), and literal keys must appear sorted, without duplicates—a formatting error otherwise, consistent with fields and declarations. iteration order is always sorted-key order, so encoding is canonical for free.

3. entries m yields entry records (fields key, value; the name is reserved). map traversal dogfoods records and constructor patterns instead of inventing tuples.

4. numeric strictness. int + float64 is an error, not a coercion; convert with to_float. floats render as 1.0, never 1; float division by zero is err, same as int. JSON numbers decode as int when written integral, float64 otherwise.

5. JSON null is json_null, not none. the honest reason: none is propagation-hostile as data—construct a record with it and the record never gets built, because propagation eats it. that's correct behavior for absence-as-failure and wrong for null-as-value, so null gets a marker type. this points at a real gavel: kanso may want zero-field types (type null with no body)—today a type requires at least one field, so the marker carries a dummy bool. it's the one visibly inelegant thing in the library.

6. the allowed-error / defect split is a word, not a sigil. ruby marks the raising variant with !; kanso would have to double every api to do that. instead, must converts any allowed failure into a defect—two lines of ordinary overloads, composing with every function ever written. parse errors from user input stay handleable; must (decode config) declares "this failing is a bug," and the defect rides the rails to the root reporter. still owed: the endpoint rule treating defect as auto-reported rather than must-be-handled.

7. small additions the work demanded: push (list accumulation), chars/char_code/from_code (the minimal unicode bridge), join, slice, string escapes \t and \r, and type-postfix brackets lexing tight (json[]) while list arguments stay spaced (f [1 2]). all prelude candidates for the import gavel.

tier 03friction—where a developer would sigh

no short-circuit and/or. we wrote both as a two-arm overload and it works, but eager evaluation means it can't guard (both (p > 0) (expensive p) runs both). candidate gavel: lazy and/or words with the same thunk mechanics if already uses.

no negative literals, no modulo. -1 is unwritable (only 0 - 1), and hex4 computes remainders by subtract-multiply. both feel like missing table stakes; both interact with the operator gavel that's already queued.

alphabetical order scatters cohesion. the sixteen tests sort into the middle of the implementation, and helper families stay adjacent only because we named them into adjacency (str_char, str_chars, str_escape...). developers will name-game the ordering rule; that's a signal. modules will absorb most of it—tests want to be a sibling file—but the rule deserves a second look with this evidence in hand.

lambdas can't destructure. encoding map entries needed a named encode_entry (entry k v) where a pattern lambda would have been one line. queued with the destructuring family.

positions blur where none propagates far. most eof arms report exact positions, but a failure that rides many frames before conversion loses locality. the fine-grained-failure story (typeset-based propagation beyond err/none) is the real fix.

the library lives at github.com/ClayShentrup/kanso-json, and runs in kanso's ci on every push—the language now has a second heartbeat.
