design note

the flow model

every program answers two questions that most languages braid into one: where does data go, and what order do things happen in? kanso keeps them apart, one construct each. once you see the split, everything about groups, walls, and dots stops being a rule to memorize and becomes the only way it could work.

the two axes

data moves through names. a binding hands a value forward; the dot hands a value into the next call. if step two needs step one's result, that need is the ordering—no operator required, and the compiler is free to run everything else at the same time.

data flow orders itself
low = cheapest prices     # these two share nothing:
total = sum prices        # they may run in parallel
report low total           # needs both -- the barrier is the data

order without data moves through >>. sometimes two effects share nothing, but you still care when they happen: serve the tea only after it steeped and the cups warmed. that ordering carries no value—it is pure sequence, and it gets its own symbol.

order that data can't express
main =
  print "steeping the sencha"   # a group: no order between members,
  print "warming the cups"      # exactly as if each ran on its own thread
  >> print "serving"            # the wall: runs after both settle

what actually crosses a wall

here is the question that keeps coming back, answered once: nothing crosses a wall except permission.

the members of a group are bare lines. nobody binds their results, so their results are discarded where they stand—the compiler even refuses a bare line whose value could never be an effect, because a value nobody can read is a bug, not a statement. when every member settles, the group as a whole carries no value at all. the wall then asks exactly one thing: did anything above me fail?

whatcrosses the wall?
a member's valueno — discarded at the member's own line, never collected
timingyes — everything above has fully settled before anything below starts
failureyes — and it is the only survivor: every member still runs to completion, then all errs merge into one, and that one err gates the wall and short-circuits from there to the top

so no—the group's results are not gathered up and threaded through the wall one by one. there is nothing to thread. the members ran for their effects; their values were never part of the story. if you find yourself wanting a member's value on the other side of a wall, the language is telling you it was never a group member at all: it is a binding, and the moment you name it, the data dependency replaces the wall.

want the values? then it's data flow, not a group
a = random 6              # bindings fan out the same way a group does --
b = random 6              # nothing shared, so these may run in parallel
print "rolled {a} and {b}"  # and the read is the join

the dot is the only place a value chooses

for the reader who wants the theory named once: . is bind—a value crosses into what comes next, and what comes next gets to look at it. >> is sequence-and-discard—run this, then that, pass nothing. and a group is the applicative layer: its shape is fixed before anything runs, no member's value can steer another member, and that is precisely why the compiler may run them in parallel without asking. the parallelism isn't an optimization it hopes is safe; it is safe by construction, because the syntax that would create a dependency doesn't exist there.

all three, one program
main =
  order = take_order          # binding: the value survives, by name
  brew order . pour           # dot: the brew's value crosses into pour
  print "irasshaimase"        # group member: value discarded, effect kept
  >> print "closing up"       # wall: carries permission, nothing else

a value crosses at a name or a dot. a wall carries permission. a group's values die where they stand, and only failure survives—merged, and moving.

the whole model in five lines

  • data crosses only at bindings and . — if downstream needs a value, name it or pipe it; the dependency is the order.
  • >> carries no data — it carries one bit: nothing above me failed.
  • a group discards every member's value — members exist for their effects; all of them always run.
  • only failure survives a group — errs merge into one and short-circuit from the wall onward. an err is a true exception: you don't handle it, and the parallel join is the one place it pauses (to collect its siblings) before resuming its exit.
  • parallel is the default because nothing can depend on anything — the group's syntax has no way to express a dependency, so the compiler doesn't have to prove independence; it is unrepresentable to violate.

everything else on this site—the wall canon, fused stages, wrap lines—is about how these constructs are written. this page is about what they mean, and the meaning fits in one breath: name it if you need it, wall it if you only need it done, and let failure find its own way out.
