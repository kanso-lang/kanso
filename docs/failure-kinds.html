design note

the two failures

a failure shows up as a hundred different things: a missing key, a dead socket, a number that won't parse. they all answer the same question, though—did the world fail you, or did your own code produce a wrong?—and that answer decides whether you handle it, where you handle it, and whether trying again could help.

assume, or handle

you, not the world, decide whether a call is one you assume works or one whose failure you handle. the same missing key can be an error you let rise or an absence you cope with, depending on what you meant by the lookup—so kanso makes you say which, in the function you reach for, instead of picking a default for you. you can't assume by accident, and you can't cope by accident.

access is a decision, not a lookup
count = m["dango"]          # i assume it's there. a miss is an err that rises.
count = at m "dango" none   # i'm choosing to cope. a miss is a none i dispatch on.

the same shape governs the wire. a dead network is rare enough that wrapping every send in coping-logic is complexity you'd pay a thousand times to catch a once—so by default you don't, and a dead network becomes exceptional precisely because you decided it wasn't worth pre-paying for. the day you want best-effort, you say so, and it's a different word:

the wire is a decision too
receipt = send bytes            # i assume this lands. non-delivery is an err.
result  = try_send bytes        # i'll cope. non-delivery comes back as a none.

this is error handling as yagni. not-handling is the correct default, because a handler for a failure that never recurs is speculative complexity. you add coping reactively, one place at a time, when a real failure earns it—the exact inverse of defensive code, which pays the tax on every line up front to guard against things that mostly never happen. the happy path stays happy not because failures can't occur, but because you refused to complicate it for failures you judged too rare to warrant it.

nothing in kanso rescues

an err is the failure you declined to handle. so it does the only sensible thing: it rises. it rides the rails out through every function that never mentioned it, carrying where it was born and everywhere it passed through, until it reaches a boundary.

an err reaching the topshipped
fn price item
  _base_price item + 50     # + 50 never runs on the sad track

main = print (price "pocky")
kanso run
error[endpoint]: unhandled err reached main: we do not serve pocky
  born in _base_price
  passed through price ← main

the rule underneath it is structural, and it needed no law at all. an arm that claims an err—fn describe (err reason)—is ordinary dispatch, writable anywhere. what keeps errors honest is what a handler can and cannot do. functions are pure and nothing is nil, so a handler that catches an err and cannot genuinely continue has nowhere to hide: it must return something, it cannot conjure a valid value from nothing, so it returns another err. the exception re-raises itself through the value model. the only handlers that can terminate a failure are the ones holding a real alternative—a default, a retry, a degraded answer—which is precisely the set of handlers that should exist. write the arm where you have one; where you don't, the err rides on without a line of plumbing.

compare the language that made "ignored the error" famous. go returns a pair, (value, error)—a product type, so four states exist, and two of them are lies. the idiomatic lie is ("", err): the error is right there beside a perfectly usable zero value, the caller grabs the value half, and the type system waves it through. errcheck exists because go's types can't say "you got one or the other." one channel dissolves this: when a kanso function fails, the err isn't beside your value, it's instead of it. there is no valid half to grab. using the result means scrutinizing it, and scrutiny has exactly two outcomes—an arm you deliberately wrote, or the ride continuing. ignoring an error isn't linted here; it's unspellable.

an err nobody claims rides to the endpoint—the executor, the top of main—and ends the program with its receipts: where it was born, everywhere it passed. that is the default fate of every unhandled failure, and it costs nothing to arrange because it's what happens when you write nothing.

policy is data-in, effect-outproposed
fn report failure              # Failure -> Description. a normal function.
  respond 500 (page (reason failure))

main = serve app . on_failure report

report takes a Failure and returns a description—an ordinary function on inert data. the endpoint hands it the failure and asks only what happens next; passing report around is harmless because it holds no special power, just an argument type. that's the shape of every policy in the language: data in, effect out, and the machinery that delivered the data stays out of your hands.

a value, or an event

here's the distinction the rest of it rests on. take two failures:

a value
n = to_int "abc"
an event
body = get "https://api/rows"

to_int "abc" does not sometimes fail. it equals an err, the way 2 + 2 equals 4. it is a constant. re-evaluating it is meaningless—you can't retry a value into being a different value, any more than you can retry 4 into being 5. get url is the opposite: it doesn't equal anything until it runs, and running it again is a genuinely new occurrence with a possibly different result. one is a value; the other is an event.

a failure's kind is fixed by where it is born—not by where its inputs came from.

this is the part that looks like a paradox and isn't. "but \"abc\" came from input—input can always change, so isn't every failure contingent?" no. trace it carefully: the input already arrived. the read succeeded; it delivered "abc" intact. the parse then failed in pure code that never touched a socket—it merely turned a value into a verdict. the provenance of "abc" is irrelevant. the failure was born in the core, so it is deterministic, so it is a value. the changeability you feel lives one step upstream, at the read—and retrying the read is a different call, not a retry of the parse.

when the wire hands back a verdict

the sharpest test is a failure that arrives over the wire and is still a value. an http 400:

same socket, two kindsproposed
fn _verdict (status 503 body)
  err (stalled body)      # the io itself faltered — a not-yet. retry may land.

fn _verdict (status 400 body)
  err (rejected body)     # the round-trip SUCCEEDED and returned a no — a wrong.

fn _verdict (status 200 body)
  body

a 400 came from io as hard as anything can. but the round-trip completed—it delivered a verdict about your request. the io didn't fail; a pure judgment on io-delivered bytes did. so 400 is a wrong—retrying ships the same rejected bytes and earns the same rejection, three times, at three times the latency. a connection-refused on the very same socket is a not-yet—the io itself couldn't complete, and the world may have moved by the next attempt. the wire is identical; the failure kind is not. the question was never "did it come over io," it was "did the io itself fail, or did a completed io hand back a value pure code then rejected?"

the compiler can see which is which

which means kanso doesn't have to ask you whether a failure is retryable. it can see it. effects are descriptions—a K_DESC the executor runs—and pure computation is values; the runtime already records where every err was born. born executing a description, at an io that couldn't complete → a not-yet, and retry is meaningful. born in pure reduction → a wrong, and retry is a category error.

retry only means something for eventsproposed
get "https://api/rows" . retry 3   # retrying an event — the only place the word means anything

to_int x . retry 3                 # retrying a value — a no-op the compiler is entitled to reject

and retry is a policy on the event, not a rescue: it re-runs on a not-yet, and lets a permanent failure rise. it never turns a failure into an unrelated value, so it needs no rescue authority—which is exactly why a library can hand it to you. it can only re-attempt and then get out of the way. the one care it demands is idempotency: re-running a get is free; re-running a charge is not, so the library retries safe methods by default and makes you assert the rest.

the whole model

a failure asks one question—did the world fail me, or did i produce a wrong?—and the rest follows from the answer. a wrong is a value: deterministic, born in the core, unit-testable, and futile to retry; you fix it by changing the code or the inputs, which is a different call. a not-yet is an event: born at the shell, contingent on a world that can move, and the only thing retry was ever for. you decide whether to assume or handle it when you write the call. and if you handle none of it, the failure rises—carrying its receipts—to the endpoint, which ends the program with the whole story in hand. handling it early is always yours to choose and never forced on you; a handler without a genuine alternative can only pass the failure along, so the truly exceptional event reaches the top under its own power.

shipped works in the compiler today: failure as a value, automatic propagation with born-in / passed-through receipts, none as dispatchable absence.
